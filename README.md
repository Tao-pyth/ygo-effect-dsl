# ygo-effect-dsl

A formal Domain-Specific Language (DSL) and state-transition framework  
for modeling and exploring Yu-Gi-Oh! card effects.

---

## Overview

`ygo-effect-dsl` is a research-oriented CORE engine that:

1. Reads normalized card data (JSONL) generated by `ygo-effect-dsl-etl`
2. Converts effect text into a structured DSL
3. Validates DSL shape against spec contracts
4. Analyzes unresolved areas to guide dictionary/spec refinement
5. Provides foundations for future state-transition exploration

This repository does **not** perform data acquisition.  
All raw and normalized data is produced by the ETL repository.

## Research Objective

The goal of this project is to:

> Formalize card effects as state transition rules  
> and enable computational exploration of playable sequences.

By converting natural language effect text into structured DSL,
we make it possible to:

- Enumerate valid effect activations from a given state
- Explore opening lines (初動展開)
- Analyze reachable board states
- Compare structural properties of decks

## Core Concepts

### 1. DSL (Effect Formalization)

Each card effect is transformed into a structured representation:

- trigger
- condition
- cost
- action
- restriction

The DSL is deterministic and machine-processable.

### 2. State Representation

A state consists of:

- Deck
- Hand
- Field
- Graveyard
- Banished
- (optional) Stack/Chain (future extension)

States are immutable snapshots.

### 3. Transition Model

Given state S:

1. Enumerate applicable effects E
2. Apply effect e ∈ E
3. Generate next state S'
4. Repeat (bounded search)

## CLI Usage

### 1) ingest

Read JSONL and report minimum raw-contract issues while continuing processing.

```bash
ygo-effect-dsl ingest examples/sample_cards.jsonl
```

### 2) transform

Convert RawCard(JSONL) to Spec v0.0 skeleton YAML.

```bash
ygo-effect-dsl transform --in examples/sample_cards.jsonl --out data/dsl_out
```

- one file per card: `{cid}.yaml`
- no missing required keys in v0.0 skeleton
- unknown values normalized as empty values (`""`, `[]`, `{}`)

### 3) validate

Validate generated DSL against v0.0 minimum grammar constraints.

```bash
ygo-effect-dsl validate data/dsl_out
```

Exit codes:
- `0`: no critical validation errors
- `1`: critical validation errors found
- `2`: argument/configuration error

### 4) analyze

Compute unresolved rates (e.g., empty object ratios) and write report.

```bash
ygo-effect-dsl analyze data/dsl_out --out data/reports
```

Output:
- `data/reports/analysis_report.json`

## Scope

Included:

- ingest/transform/validate/analyze pipeline
- DSL generation and contract validation
- Research experiments around formalization
- Foundations for state-transition modeling

Excluded (for now):

- API fetching
- image downloading
- direct SQLite dependency on ETL internals
- full chain/stack simulation
- opponent interaction modeling

---

## Limitations

- Current DSL transformation is skeleton-first (structure priority over deep semantics)
- No opponent interaction in v0.0 execution model
- No strict chain resolution in current core implementation
