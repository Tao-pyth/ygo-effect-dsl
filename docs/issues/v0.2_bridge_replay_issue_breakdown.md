# V0.2 Bridge / Replay Issue Breakdown

Status: planning baseline

## Parent Roadmap Issue

Title: `Roadmap: V0.2 Bridge / Replay baseline`

- 決定事項: Python はルールを持たない。Bridge は変換責務のみ担当する。Replay は再現性のみ担当する。
- 未決事項: DecisionRequest 正式仕様、正式 Action schema、ocgcore 実接続方法。
- 依存Issue: なし。
- 子Issue: D1-D6、I1-I7。
- 受け入れ条件: Bridge mock、Replay JSON round-trip、仕様文書、smoke test が揃う。

## Design Issues

### D1: Action-like dict と正式Action modelの境界を定義する

- 決定事項: V0.2 の Action-like dict は正式な探索用 Action ではない。
- 未決事項: 正式 Action ID、Action schema、探索用粒度。
- 依存Issue: D4。
- 子Issue: なし。
- 受け入れ条件: `docs/bridge/messages.md` と将来 Action model の関係が説明される。

### D2: DecisionRequestのV0.2非目標とV0.3以降の入力を整理する

- 決定事項: V0.2 では DecisionRequest を完全仕様化しない。
- 未決事項: request signature、選択肢列挙、hidden information の扱い。
- 依存Issue: D4。
- 子Issue: なし。
- 受け入れ条件: V0.3 で仕様化すべき入力と V0.2 で扱わない範囲が明記される。

### D3: ReplayHistory / ActionRecordの最小仕様を確定する

- 決定事項: `schema_version`、`seed`、`deck_identifier`、`actions`、`version_metadata`、`notes` を最小必須フィールドとする。
- 未決事項: snapshot、clone_duel、random choice trace。
- 依存Issue: なし。
- 子Issue: I1、I2、I3。
- 受け入れ条件: `docs/replay/format.md` と実装が一致する。

### D4: Bridge Message / Action-like / Core inputの変換契約を確定する

- 決定事項: Message dict から Action-like dict、Action-like dict から core input dict の二方向のみ扱う。
- 未決事項: ocgcore binary decode、全 request type、invalid response 時の duel 継続方針。
- 依存Issue: なし。
- 子Issue: I4、I5、I6。
- 受け入れ条件: `docs/bridge/messages.md` と mock adapter の変換が一致する。

### D5: Deterministic replay guaranteeを定義する

- 決定事項: JSON serializer は UTF-8、`ensure_ascii=False`、`indent=2`、`sort_keys=True`、末尾改行ありとする。
- 未決事項: core version、Lua version、cards.cdb の厳密照合。
- 依存Issue: D3。
- 子Issue: I2、I7。
- 受け入れ条件: round-trip test で同一 dict と deterministic output を確認する。

### D6: ocgcore実接続調査の観点を整理する

- 決定事項: V0.2 では実 ocgcore 接続を実装しない。
- 未決事項: dynamic library loading、EDOPro Lua 実行、cards.cdb 読み込み、実 duel 進行。
- 依存Issue: D4。
- 子Issue: なし。
- 受け入れ条件: V0.3 以降で調査すべき接続観点が列挙される。
